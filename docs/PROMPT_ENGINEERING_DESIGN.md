# AI 프롬프트 설계서 — 프롬프트 학습 평가 시스템

> 본 문서는 시스템 내에서 실제로 동작하는 AI 프롬프트들의 설계 근거와 엔지니어링 결정을 정리한 문서입니다.

---

## 전체 AI 프롬프트 구조

이 시스템은 5개의 독립적인 LLM 호출 모듈로 구성됩니다. 각 모듈은 서로 다른 온도, 출력 형식, 역할을 갖습니다.

```
학생이 프롬프트를 작성하고 제출하는 흐름

  [결과 실행 버튼]
      └─► RubricEvaluator × 1회    루브릭 기준 점수 산출 (미리보기)
      └─► PromiCoachService         실행 결과 기반 맥락 코칭

  [최종 제출 버튼]
      └─► RubricEvaluator × 5회 평균    최종 점수 산출 (편차 흡수)

  [마이크 구두 검증 — 점수 80점 이상 통과 후]
      └─► 마이크 LLM 평가 × 문항 수    각 확인 질문 독립 평가 (70점 기준)

  [모든 마이크 문항 통과 후에만]
      └─► FeedbackAgent             최종 성장 중심 피드백 생성
      └─► LLMAnalyzer               사고력 14지표 분석 → 위험도 입력
```

## 1. 루브릭 평가 프롬프트 — RubricEvaluator

**파일:** `packages/llm_analysis/rubric_evaluator.py`

### 무엇을 하는가

학생이 제출한 프롬프트를 문제별 루브릭 기준으로 0~100점으로 채점합니다. 미리보기 실행 시 1회, 최종 제출 시 5회 반복 후 평균을 냅니다.

### 시스템 프롬프트 구조

```python
system_prompt = f"""당신은 프롬프트 엔지니어링 교육 전문가입니다.
학생이 작성한 프롬프트를 아래 루브릭 기준으로 평가하세요.

평가 가이드라인: {guidelines}

루브릭 기준:
{criteria_str}   ← 문제별 rubric_json 동적 주입

반드시 JSON 형식으로만 응답하세요:
{{
  "criteria_scores": [
    {{"name": "기준명", "score": 0~max_score, "feedback": "구체적 피드백"}}
  ],
  "overall_feedback": "전체 평가 요약",
  "strengths": ["강점1", "강점2"],
  "improvements": ["개선점1", "개선점2"]
}}"""
```

### 설계 결정과 이유

**temperature = 0.3** — 채점은 일관성이 생명입니다. 같은 프롬프트를 5번 평가할 때 점수 편차가 크면 신뢰를 잃습니다. 낮은 온도로 재현성을 확보합니다.

**response_format json_object 강제** — 교육 시스템에서 LLM 응답 파싱 실패는 곧 학생에게 오류 화면 노출입니다. json_object 모드로 항상 유효한 JSON이 반환되도록 강제합니다.

**rubric_json 동적 주입** — 18개 문제가 각각 다른 평가 기준을 가집니다. 프롬프트에 하드코딩하지 않고 문제별 `rubric_json`을 런타임에 주입해 하나의 평가기로 모든 문제를 처리합니다.

**5회 반복 평균** — LLM은 동일 입력에도 확률적으로 다른 점수를 냅니다. 5회 평균을 내면 통계적으로 편차가 흡수됩니다. 단, 미리보기는 1회만 실행해 토큰 비용을 아낍니다.

**총점 계산 방식** — `(score / max_score) × weight × 100`으로 기준별 가중 평균을 내 100점 만점으로 환산합니다.

### OpenAI API 없을 때 fallback

API 키가 없거나 호출에 실패하면 키워드 휴리스틱 mock 평가로 자동 전환됩니다. 개발 환경에서도 시스템이 멈추지 않도록 하는 방어 설계입니다.

```python
_CRITERION_KEYWORDS = {
    "명확성":    (["명확", "구체", "정확", ...], ["모르", "그냥", ...]),
    "역할 정의": (["역할", "전문가", "당신은", ...], []),
    "출력 형식": (["형식", "목록", "JSON", ...], []),
    "예시 포함": (["예시", "예를 들면", "예제", ...], []),
    ...
}
```

## 2. 사고력 분석 프롬프트 — LLMAnalyzer

**파일:** `packages/llm_analysis/analyzer.py`, `packages/llm_analysis/prompts.py`

### 무엇을 하는가

학생의 프롬프트 제출물에서 학습 사고 과정을 14개 차원으로 정량화합니다. 이 결과는 위험도 분석 파이프라인의 입력으로 들어갑니다.

### 시스템 프롬프트 (THINKING_ANALYSIS_PROMPT)

```
당신은 학습자의 사고 구조를 분석하는 교육 AI 전문가입니다.
아래 학생의 프롬프트를 읽고 사고 역량 14개 항목을 0.0~1.0으로 채점하세요.

 1. problem_understanding_score   문제를 정확하게 이해했는가
 2. problem_decomposition_score   문제를 하위 단계로 분해했는가
 3. constraint_awareness_score    제약 조건을 인식했는가
 4. validation_awareness_score    결과를 검증하려는 의식이 있는가
 5. improvement_prompt_score      프롬프트를 개선하려 시도했는가
 6. self_explanation_score        자신의 풀이 과정을 설명했는가
 7. reasoning_quality_score       추론의 논리적 품질
 8. reflection_depth_score        학습 과정에 대한 반성의 깊이
 9. error_analysis_score          오류 원인 분석
10. debugging_quality_score       오류 체계적 수정
11. decision_reasoning_score      결정 근거 명확성
12. approach_selection_score      적절한 접근 방식 선택
13. improvement_consistency_score 반복 개선의 일관성
14. iteration_quality_score       반복 시도의 품질 향상
```

### 설계 결정과 이유

**temperature = 0.2** — 사고력 분석은 위험도 점수의 입력값이 됩니다. 매번 다른 분석이 나오면 위험도도 흔들립니다. 가장 낮은 온도로 재현성을 극대화합니다.

**14개로 세분화한 이유** — 단일 점수로는 어떤 유형의 탈락 위험인지 알 수 없습니다. 14개 세부 지표를 통해 인지적 문제(이해·분해)인지, 반성 부재인지, 오류 분석 역량 부족인지를 구별할 수 있습니다.

**thinking 가중치를 0.10으로 낮게 설정한 이유** — LLM 분석은 행동 데이터보다 오류 가능성이 높습니다. 전체 위험도에서 10%만 반영하고 나머지 90%는 실제 행동 데이터(성과·진도·참여·프로세스)에 의존합니다.

### 5개 차원 위험도 가중치 설계

```python
BASE_RISK_WEIGHTS = {
    "performance": 0.30,  # 실제 점수 → 가장 직접적인 지표
    "progress":    0.25,  # 제출 간격, 중도 포기 → 탈락 선행 신호
    "engagement":  0.20,  # 로그인, 세션 시간 → 참여도
    "process":     0.15,  # 시도 횟수, 전략 변경 → 학습 과정
    "thinking":    0.10,  # LLM 사고력 분석 → 보조 지표
}
```

## 3. 프롬이 코칭 프롬프트 — PromiCoachService

**파일:** `apps/backend/app/services/promi_coach_service.py`

### 무엇을 하는가

학생이 문제를 풀면서 실시간으로 코칭을 받습니다. 프롬이라는 강아지 캐릭터가 단계마다 다른 관점의 코칭을 제공합니다. 핵심은 **절대 정답을 주지 않는다**는 원칙입니다.

### 3단계 모드 분기

```
enter 모드  문제를 처음 열었을 때
            → "역할, 목표, 출력 형식 세 가지부터 맞추면 좋겠어요"
            → 방향 제시, 아직 아무것도 모르는 상태

run 모드    결과 실행 버튼 클릭 후 (latest_response 포함)
            → 실제 모델 응답을 보고 "이 응답에서 아쉬운 점은..."
            → 결과 기반 구체적 수정 방향

final 모드  최종 제출 전
            → 형식·제약·예외 처리 누락 점검
            → "이것만 확인하고 내면 될 것 같아요"
```

### 시스템 프롬프트 핵심 구조

```
[역할] 당신은 '프롬이'라는 강아지 코치입니다.
[제약] 절대 정답, 완성 답안, 직접적인 해결 문장을 제공하지 마세요.
       학생이 스스로 사고할 수 있도록 방향만 제시하세요.
[출력] {
  "message": "핵심 코칭 1~2문장",
  "checkpoints": ["점검 포인트1", "점검 포인트2", "점검 포인트3"],
  "encouragement": "짧은 응원",
  "caution": "직접 답변 유도 방지 주의 문구 또는 null"
}
```

### 설계 결정과 이유

**temperature = 0.6** — 루브릭 평가(0.3)보다 높게 설정했습니다. 매번 새로운 관점의 코칭 메시지가 나와야 학생이 반복해서 실행 버튼을 눌러도 의미 있는 피드백을 받을 수 있습니다.

**max_tokens = 500 제한** — 코칭은 간결해야 합니다. 긴 설명은 학생의 인지 부담을 높이고 스스로 생각하지 않게 만듭니다.

**latest_response를 입력에 포함하는 이유** — "프롬프트를 잘 써야 해요"처럼 추상적인 조언이 아니라, "이 응답에서 출력 형식이 지정되지 않아서 ~~한 결과가 나온 것 같아요"처럼 실제 결과를 근거로 코칭할 수 있습니다.

**정답 제공 금지 원칙** — 코치가 정답을 주면 학생은 이해 없이 복사합니다. 이 원칙을 시스템 프롬프트 수준에서 명시적으로 강제합니다.

**컨텍스트 길이 제한 적용**
```python
system_prompt   최대 1,200자
user_template   최대 800자
latest_response 최대 1,000자
```

## 4. 피드백 에이전트 프롬프트 — FeedbackAgent

**파일:** `packages/llm_analysis/feedback_agent.py`

### 무엇을 하는가

마이크 구두 검증까지 모두 통과한 학생에게만 최종 성장 피드백을 생성합니다. 이전 제출 이력과 현재 점수를 비교해 성장 스토리를 만듭니다.

### 감정 상태 분류 설계

```python
if total_score >= 85:   emotion = "excited"      # 최고 성과 — 폭발적 응원
elif total_score >= 75: emotion = "happy"         # 좋은 성과 — 기쁨
elif total_score >= 50: emotion = "encouraging"   # 보통 — 동기 부여
elif total_score >= 30: emotion = "thinking"      # 미흡 — 사고 촉진
else:                   emotion = "concerned"     # 낮음 — 걱정과 응원
```

6가지 감정 상태는 UI에서 프롬이 SVG 캐릭터 표정으로 시각화됩니다. 점수를 숫자로만 보는 것보다 캐릭터 감정과 함께 전달하면 학생의 감정 연결과 동기 부여 효과가 높아집니다.

### 성장 노트 생성 로직

```python
if score_delta > 5:
    growth_note = f"지난 제출보다 {delta}점 올랐어요! 성장이 느껴져요"
elif score_delta < -5:
    growth_note = f"이번엔 조금 낮아졌지만, 다음엔 분명 더 잘할 수 있어요"
else:
    growth_note = "지난 제출과 비슷한 수준이에요. 조금 더 다듬어볼까요?"
```

### 설계 결정과 이유

**마이크 통과 후에만 호출하는 이유** — 마이크 검증을 통과하지 못한 학생에게 최종 피드백을 보여주면, 마이크 검증을 우회 수단으로 여기게 됩니다. 게다가 불필요한 LLM 호출로 토큰이 낭비됩니다.

**이전 이력 최대 3개만 사용하는 이유** — 너무 오래된 제출 이력은 현재 상태와 관련이 낮습니다. 최근 3개 제출을 기준으로 성장 방향을 파악합니다.

**temperature = 0.7** — 격려 메시지는 매번 다르게 나와야 반복해서 봐도 신선합니다. 평가(0.3)보다 높은 온도로 창의적인 메시지 생성을 허용합니다.

## 5. 마이크 개념 확인 평가 프롬프트

**파일:** `apps/backend/app/api/routes/student.py` → `evaluate_concept_reflection`

### 무엇을 하는가

학생이 마이크로 직접 말하면 음성이 텍스트로 변환되어 제출됩니다. 이 텍스트를 LLM이 평가해 학생이 개념을 진짜로 이해했는지 검증합니다. 확인 질문은 문제별로 보통 5개이며, 모든 질문에서 70점 이상을 받아야 통과입니다.

### 이 기능을 만든 이유

학생이 ChatGPT에 문제를 그대로 물어보고 답변을 붙여넣기만 해도 텍스트 평가는 통과할 수 있습니다. 음성으로 직접 설명하게 하면 복사·붙여넣기가 불가능하고, 진짜로 이해한 사람만 말로 설명할 수 있습니다.

### 프롬프트 입력 구성

```
[학생이 실제 제출한 프롬프트 전문]
[이 문제의 핵심 개념 목록]
[현재 평가 중인 확인 질문]
[학생이 마이크로 말한 내용 — 음성을 텍스트로 변환한 답변]
```

학생 제출 프롬프트를 함께 넣는 이유는, "이 개념이 본인 프롬프트 어디에 적용됐는지"까지 설명할 수 있어야 진정한 이해로 보기 때문입니다. 단순 암기가 아니라 연결 능력까지 평가합니다.

### 설계 결정과 이유

**문항별 독립 LLM 호출** — 하나의 LLM 호출에 모든 질문을 묶으면 앞 질문의 답이 뒤 질문 채점에 영향을 줍니다. 각 질문을 독립적으로 평가해 공정성을 확보합니다.

**70점 임계값** — 완벽한 설명을 요구하면 학생 부담이 과도합니다. 기본적인 이해 수준(70점)을 기준으로 해 너무 엄격하지 않되 의미 없는 답변은 걸러냅니다.

**전 문항 통과 조건** — 5개 중 4개만 통과해도 된다면, 잘 모르는 개념은 그냥 건너뛰게 됩니다. 모든 문항 통과로 완전한 이해를 확인합니다.

**`_is_concept_reflection_complete()` 재검증** — 과거에 일부 문항만 통과해도 전체 통과로 저장되는 버그가 있었습니다. 조회할 때마다 저장된 결과를 재검증해 잘못 저장된 데이터도 올바르게 처리합니다.

## 6. 공통 설계 원칙

### 역할-과제-제약-출력 4요소 구조

모든 시스템 프롬프트는 4요소로 구성됩니다.

```
[역할] 당신은 프롬프트 엔지니어링 교육 전문가입니다.
[과제] 학생 프롬프트를 아래 루브릭 기준으로 평가하세요.
[제약] JSON 형식으로만 응답하세요. 추측하지 마세요.
[출력] {"criteria_scores": [...], "overall_feedback": "..."}
```

### 용도별 temperature 분리

| 모듈 | temperature | 이유 |
|------|-------------|------|
| 루브릭 평가 | 0.3 | 일관된 채점 |
| 사고력 분석 | 0.2 | 높은 재현성 |
| 프롬이 코칭 | 0.6 | 매번 새로운 관점 |
| 피드백 생성 | 0.7 | 창의적 격려 메시지 |

### JSON 출력 강제

모든 LLM 호출에 `response_format={"type": "json_object"}` 적용합니다. 교육 시스템에서 응답 파싱 실패는 곧 학생에게 오류 화면 노출입니다.

### Fallback 계층

```
LLM 호출 (OpenAI API)
    → 실패 시 키워드 휴리스틱 Mock 평가 자동 전환
    → 개발/테스트 환경에서 API 키 없어도 시스템 정상 동작
```

## 7. 토큰 최적화 전략

### 핵심 게이트: 마이크 통과 전 /feedback 호출 차단

가장 큰 토큰 절약 포인트입니다. 마이크 검증을 통과하지 못한 학생에게는 FeedbackAgent를 호출하지 않습니다. 미통과 학생이 많을수록 절약 효과가 커집니다.

### DB 캐싱으로 재호출 방지

최종 제출 시 5회 루브릭 평가 결과를 `rubric_evaluation_json` 컬럼에 저장합니다. `/feedback` 재요청 시 DB에서 바로 꺼내 LLM을 다시 호출하지 않습니다.

### 미리보기 1회 vs 최종 5회 분리

미리보기는 탐색용이므로 1회만 평가합니다. 학생이 미리보기를 10번 눌러도 10번의 단일 호출입니다. 5회 평균은 최종 제출 때만 적용합니다.

### 경량 모델 전면 적용

gpt-4o 대신 gpt-4o-mini를 전면 사용합니다. 비용이 약 10분의 1로 줄어들어 5회 반복 평가와 실시간 코칭을 경제적으로 운영할 수 있습니다.

### 컨텍스트 길이 제한

프롬이 코칭 시 긴 시스템 프롬프트와 사용자 템플릿을 그대로 넘기지 않고 최대 길이로 잘라 불필요한 토큰 소비를 방지합니다.
